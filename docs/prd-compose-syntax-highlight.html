PRD: compose-highlight

Product Requirements Document — Android Syntax Highlighting Library — April 2026

← Home
A Kotlin Multiplatform-ready Android library that provides real syntax highlighting for Jetpack Compose using Highlight.js, the same engine used by Claude, Perplexity, and ChatGPT in production.

Machine-readable version: prd-compose-syntax-highlight.md (Markdown, optimized for AI agent consumption)

1. Problem Statement

Android apps that display source code need syntax highlighting. The current options are all flawed:

  • WebView-only rendering — loses native text selection, accessibility, theming integration, and Compose interop
  • Regex-based tokenizers — fragile, limited language support, impossible to maintain
  • Custom parsers per language — enormous effort, never reaches parity with established libraries

Production apps (Claude, Perplexity, ChatGPT) have converged on a hidden-WebView pattern: run Highlight.js off-screen for tokenization, then convert the output to native spans. But each app re-implements this from scratch. compose-highlight extracts this proven pattern into a reusable library with a single-line Compose API.

2. Background: Production Evidence

This design is based on reverse-engineering five production Android APKs:

AppEngineVersionLanguagesRendering
ChatGPTHighlight.jsv11.9.041WebView
ClaudeHighlight.jsv11.10.0192Native Compose spans
PerplexityHighlight.jsv11.11.1192Native Compose spans
GitHubServer-sideN/AAllNative StyleSpans
GeminiNone (shell)N/AN/AN/A
Key insights from production:
  1. Perplexity produces both light AND dark AnnotatedStrings from a single JS call — instant theme switching
  2. Perplexity parses CSS theme files at runtime into a LinkedHashMap<String, SpanStyle> — any hljs theme works
  3. Claude uses Base64 data: URLs; Perplexity uses WebViewAssetLoader (cleaner, recommended by Google)
  4. Both Claude and Perplexity cache the WebView singleton and reuse it across all calls
  5. Perplexity uses jsoup for HTML parsing and recursively walks spans into Compose AnnotatedString
  6. Perplexity wraps failures in MarkdownException$HighlightJsFailed and falls back to unstyled code

<script>
function highlightCode(code, lang) {
  const block = document.getElementById("code");
  block.textContent = code;
  block.className = lang;
  delete block.dataset.highlighted;
  hljs.highlightElement(block);
  return block.innerHTML;
}
</script>

4.2 HighlightEngine

Core class managing the hidden WebView and JS execution. Modeled after Perplexity's ra/d and ra/c.

class HighlightEngine internal constructor(context: Context) {
    suspend fun initialize()
    suspend fun highlightToHtml(code: String, language: String): Result<String>
    suspend fun highlight(code: String, language: String, theme: HighlightTheme): Result<AnnotatedString>
    suspend fun highlightBothThemes(
        code: String, language: String,
        lightTheme: HighlightTheme, darkTheme: HighlightTheme,
    ): Result<ThemedHighlightResult>
    fun destroy()
}

data class ThemedHighlightResult(val light: AnnotatedString, val dark: AnnotatedString)

WebView init sequence:

  1. Create WebView(context) on Main thread
  2. Enable JavaScript: settings.javaScriptEnabled = true
  3. Create WebViewAssetLoader with AssetsPathHandler
  4. Set WebViewClient — onPageFinished() completes readyDeferred
  5. Load https://appassets.androidplatform.net/assets/compose-highlight/bridge.html
  6. Cache WebView instance for reuse

JS execution:

  1. Escape: ' → \', \n → \\n, \r → \\r
  2. Build: (function() { return highlightCode('ESCAPED', 'LANG'); })()
  3. evaluateJavascript(js, callback)
  4. Unescape response: < → <, \" → ", \\n → LF

4.3 ThemeParser

Parses hljs CSS themes at runtime. Modeled after Perplexity's gn/d.c().

object ThemeParser {
    fun parse(context: Context, cssAssetPath: String): Map<String, SpanStyle>
    fun parse(cssText: String): Map<String, SpanStyle>
}

Extracts color, font-weight, font-style, and background from CSS rules. Maps .hljs-* class selectors to Compose SpanStyle instances. Results are cached via Lazy.

4.4 HtmlToAnnotatedString

Recursive HTML span walker. Modeled after Perplexity's gn/d.b().

object HtmlToAnnotatedString {
    fun convert(html: String, colorMap: Map<String, SpanStyle>): AnnotatedString
}

Uses jsoup to parse HTML. Walks the DOM tree recursively: for each <span class="hljs-*">, pushes a SpanStyle from the color map, recurses into children, then pops. Text nodes are appended verbatim.

8. Threading Model

Caller coroutine (any dispatcher)
    │
    ├─ mutex.withLock {                    // Serialize calls
    │      ├─ withContext(Dispatchers.Main) // WebView requires Main
    │      │      evaluateJavascript(js, callback)
    │      └─ suspendCancellableCoroutine  // Bridge callback → coroutine
    │
    ├─ ThemeParser.parse()                 // Pure computation, any thread
    ├─ HtmlToAnnotatedString.convert()     // Pure computation, any thread
    └─ Return AnnotatedString

Critical: WebView MUST be created and accessed on Main thread. Use Mutex to serialize concurrent evaluateJavascript() calls. Do NOT create multiple WebViews.

9. Error Handling

Follow Perplexity's pattern: graceful degradation to unstyled code.

ScenarioHandling
WebView fails to initResult.failure(HighlightException.WebViewInitFailed)
JS execution errorResult.failure(HighlightException.JsExecutionFailed)
Unknown languageReturns un-highlighted HTML (no crash)
CSS theme not foundResult.failure(HighlightException.ThemeNotFound)
HTML parse failureResult.failure(HighlightException.HtmlParseFailed)
Timeout (>5s)Result.failure(HighlightException.Timeout)
 
sealed class HighlightException(message: String, cause: Throwable? = null) : Exception(message, cause) {
    class WebViewInitFailed(cause: Throwable) : HighlightException(...)
    class JsExecutionFailed(cause: Throwable) : HighlightException(...)
    class ThemeNotFound(path: String) : HighlightException(...)
    class HtmlParseFailed(cause: Throwable) : HighlightException(...)
    class Timeout : HighlightException(...)
}

Theme color mapsLazy initParsed once (~1-2ms), cached forever

Highlight resultsCaller caches via remember {}Library doesn't know caching policy

Expected latency: First call ~200-400ms (WebView init), subsequent ~10-50ms (JS + parsing). Theme parsing ~1-2ms (one-time).

12. Non-Goals

  • Language auto-detection — user must specify language
  • Diff highlighting — no added/removed line coloring
  • Code editing — read-only display only
  • Custom language grammars — use what Highlight.js bundles
  • KMP (initial version) — Android-only; pure-Kotlin layers can be extracted later
  • Android View support — Compose only; use ComposeView for legacy
  • Font bundling — uses system monospace; users pass custom FontFamily

  1. Project setup — Gradle project, dependencies, build config
  2. Asset bundle — Download highlight.min.js, create bridge.html, add theme CSS
  3. ThemeParser — Implement + unit test (no Android deps)
  4. HtmlToAnnotatedString — Implement + unit test (jsoup only)
  5. HighlightEngine — WebView manager + JS bridge + instrumented tests
  6. HighlightTheme — Wire up lazy theme maps
  7. SyntaxHighlightedCode — Full composable with all UI features
  8. HighlightThemeProvider — CompositionLocal + auto dark/light

  10. README — Usage docs, API reference

15. Success Criteria

  • SyntaxHighlightedCode(code, language) renders colored code with a single line
  • All 192 Highlight.js languages produce highlighted output
  • Light/dark themes switch instantly without re-highlighting
  • Code is natively selectable (long-press)
  • Copy button copies raw unstyled code
  • Line numbers display and align correctly
  • Horizontal scroll for long lines
  • Unknown language falls back to unstyled monospace
  • WebView failure falls back to unstyled monospace
  • Custom CSS themes work by dropping a file into assets
  • All unit + instrumented tests pass
  • Sample app demonstrates all features
  • Library size overhead <1.5 MB
